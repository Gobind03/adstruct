# Architecture

## Overview

The Marketing Suite is an AI-driven digital marketing platform for managing conversational ad campaigns across platforms like ChatGPT and Perplexity. It follows a monorepo structure with clearly separated backend and frontend applications.

## System Architecture

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Angular    │────▶│  Spring Boot │────▶│  PostgreSQL  │
│   Frontend   │◀────│   Backend    │◀────│   Database   │
│   (4200)     │     │   (8080)     │     │   (5432)     │
└──────────────┘     └──────────────┘     └──────────────┘
```

## Backend Modules

The backend follows a modular monolith pattern organized by domain:

| Module | Package | Purpose |
|--------|---------|---------|
| Identity | `identity` | Organizations, workspaces, users, memberships, audit |
| Integration | `integration` | Integrations Hub: 28 platform providers, OAuth2 flows, connectors, sync jobs, webhook receive pipeline, entity mappings, campaign report data, health |
| Campaign | `campaign` | Conversation campaigns, targeting, sponsored units, approvals, synced metrics from integrations |
| Measurement | `measurement` | Ad events ingestion and analytics |
| Creative | `creative` | Creative Studio: assets, copy artifacts, variants, usage, links, AI generation, folders |
| Auth | `auth` | JWT authentication and token management |
| Common | `common` | Shared utilities, exceptions, security, audit aspect |

### Layered Architecture (per module)

```
api/        → Controllers, DTOs (records), MapStruct mappers
service/    → Business logic, transactional boundaries
domain/     → JPA entities, enums
repo/       → Spring Data JPA repositories
```

## Key Design Decisions

### Authentication & Authorization
- **JWT-based stateless auth** with access + refresh tokens
- **RBAC** with roles: ADMIN, EDITOR, ANALYST, APPROVER
- Membership scoped to organization and optionally to workspace
- `@PreAuthorize` annotations on controller methods

### Data Layer
- **PostgreSQL 16** with JSONB columns for flexible schema fields
- **Flyway** for versioned database migrations
- **UUID primary keys** generated by the database
- `OffsetDateTime` for all timestamps (timezone-aware)

### DTO Strategy
- Java records for immutable DTOs
- MapStruct for entity ↔ DTO mapping
- Jakarta Validation on request DTOs

### Error Handling
- RFC 7807 Problem Details for all error responses
- Global exception handler with typed exceptions
- Correlation IDs for request tracing

### Audit Logging
- AOP-based audit logging via `@Auditable` annotation
- Before/after JSON snapshots stored in `audit_logs` table

### Secret Management
- `SecretStore` interface with pluggable implementations
- `LocalSecretStore` for dev (AES-GCM encrypted in-memory)
- Placeholder for AWS Secrets Manager in production

### Observability
- Structured JSON logging (Logstash encoder) in non-dev
- Correlation ID propagation via MDC
- Spring Actuator metrics exposed at `/actuator`

## AI Platform Layer

The suite includes a first-party **AI platform** for workspace chat, approved prompt runs, workflows, provider configuration, safety policy, and human-in-the-loop **action proposals**. Backend code lives under `com.avyukt.marketsuite.ai`; HTTP routes are documented in [api.md](api.md#ai-platform-api) and the conceptual guide is [ai.md](ai.md).

### Data flow: chat turn, tool calls, and proposals

1. **Chat turn**: Client `POST .../ai/conversations/{id}/messages` → `AiConversationService.postMessage` loads history, resolves `AiProviderConfig` (and model), optionally runs **deterministic tools** in `TOOL_ASSISTED` mode, builds `LlmMessage` list → `LlmGatewayRouter` → assistant message persisted after **redaction**.
2. **Tool calls**: Tool metadata is served from `GET .../ai/tools`. Execution paths are implemented in services with workspace policy (`allowedTools` in safety policy JSON) constraining what may run. Model-facing tool digests are attached to context; risky mutations should surface as **proposals** instead of silent writes.
3. **Proposals**: `AiActionProposal` rows move `PROPOSED` → `APPROVED` / `REJECTED` → optional `EXECUTED` / `FAILED` via `AiActionProposalsController`, isolating side effects behind reviewer and executor permissions.

### Integration points with other modules

- **Identity / permissions**: All AI endpoints use `PermissionService` (e.g. `requireAiUse`, `requireAiOrgManagement`, `requireAiActionApproval`).
- **Integrations**: `AiProviderConfig` references `IntegrationAccount` for encrypted credentials (`secretRef` → `SecretStore`).
- **Governance**: `AiRewriteController` under `/workspaces/{id}/governance/ai` uses the AI stack to rewrite copy against brand/compliance context (see [api.md](api.md#ai-platform-api)).
- **Cross-module API**: Other services inject **`AiFacade`** for approved prompt runs, conversations, messages, and programmatic proposal creation (see [ai.md](ai.md)).

## Brand & Content Governance

The Governance module centralizes brand identity, compliance rules, disclaimers, and content templates across the suite.

### Inheritance Model

- **OrgBrandProfile** defines organization-level defaults (colors, fonts, voice tone, languages)
- **WorkspaceBrandProfile** stores a JSONB `overridesJson` that selectively overrides org defaults
- The `getEffectiveProfile()` service method performs a deterministic merge: org base fields + workspace overrides (overrides win)

### Rule Engine

- Deterministic, no LLM calls
- Scans content against `BrandRule` entries within a `BrandRuleSet`
- Rule types: BANNED_PHRASE (string contains + regex), REQUIRED_DISCLAIMER, CLAIM_RESTRICTION, platform constraints
- Severity aggregation: any BLOCK finding -> FAIL, any WARN -> WARN, else PASS
- Results stored as `GovernanceCheckRun` with JSONB `findingsJson`

### Approval Flow

- Templates follow DRAFT -> IN_REVIEW -> APPROVED/REJECTED -> ARCHIVED lifecycle
- Submitting a template for review creates an `ApprovalWorkflow` row (reuses existing entity from campaign module)
- Only users with APPROVER, WORKSPACE_ADMIN, or ORG_ADMIN roles can approve/reject
- Approved templates can spawn new versions via parentTemplateId chain

### How research connects to other modules

- **Identity / audit**: Membership and `AuditService` for publish, AI actions, and entity changes.
- **Integrations**: `ResearchSource` may reference `integration_account_id` / `integration_resource_id` for synced assets.
- **Governance**: Weekly digest workflow invokes `governance.getEffectiveBrandProfile`; brand context feeds `research.weekly_digest_narrative`.
- **Campaigns & content**: `ResearchLink` maps research entities to `CAMPAIGN`, `TEMPLATE`, `RULESET`, `BRAND_ASSET`, etc., for traceability from campaigns back to evidence.
- **AI platform**: Approved prompt templates and workflow definitions; `AiPromptRun` rows for each LLM step; optional AI tools (`research.listRecentSnapshots`, `research.getSnapshot`, …).

### Provenance chain: Source → Snapshot → Evidence → Insight

1. A **research source** (URL, file, integration, note) is created under the workspace.
2. **Snapshots** capture content at a point in time (`SnapshotType` describes the capture shape).
3. **Insights** start as `DRAFT`; **insight_evidence** rows link an insight to snapshots with citation text.
4. **Publish** moves an insight to `PUBLISHED` only if at least one evidence row exists and the actor has `requireResearchPublish` (org admin or workspace admin).

### AI integration flow: Snapshot → AiFacade → AiPromptRun → ResearchAiRunLink → Artifact

1. User or job triggers an operation (e.g. summarize, extract, cluster, persona draft).
2. `ResearchAiService` loads snapshots, builds JSON input, calls **`AiFacade.runPrompt(workspaceId, templateName, inputJson, …)`** (weekly digest uses **`AiWorkflowService.run`** with `research.weekly_digest`).
3. The AI layer persists an **`AiPromptRun`** (output JSON/text, model metadata).
4. **`ProvenanceService`** validates citation snapshot IDs where applicable, then persists **`ResearchAiRunLink`** with `produced_entity_type` (`SNAPSHOT_SUMMARY`, `INSIGHT_DRAFT`, `KEYWORD_CLUSTER`, `PERSONA_RESEARCH`, `DIGEST_REPORT`), `produced_entity_id`, and JSON **`snapshot_ids`** for inputs used.
5. The service creates or updates the **artifact** (snapshot summary fields, insights + evidence, clusters, persona, digest report).

See [ai.md](ai.md#how-research-module-uses-aifacade) for the `AiFacade` pattern and citation validation behavior.

### How Creative Studio connects to other modules

- **Identity / permissions**: `PermissionService` — `requireCreativeRead`, `requireCreativeManagement`, `requireCreativeAiUse` (see [creative.md](creative.md#rbac)).
- **Governance**: `GovernanceCheckService` on `POST .../copy/{id}/governance-check` and after AI copy generation when a rule set is supplied; `governance_check_run_id` on `creative_copy_artifacts`.
- **Research**: AI inputs may reference persona, keyword cluster, and insight IDs; citations in generated ad copy variants.
- **Campaign**: `sponsored_units` optional `copy_artifact_id` / `asset_id`; `creative_usages` for broader usage tracking.
- **AI platform**: Approved templates `creative.generate_*` / `creative.enrich_asset_metadata`; `AiPromptRun` rows; enrichment creates an **action proposal** for approval before applying metadata changes.

Controllers follow the same layered pattern as other modules: `api/` → `service/` → `domain/` → `repo/`.
